# Token-Boundary Remediation

Native components should consume public `--ps-*` semantic and component tokens; PrimeNG `--p-*`
tokens belong in the provider preset or adapter boundary, not in a component that claims to be
provider-independent. This backlog tracks the remediation of that debt across the native component
set (finding-tracked, see `TOKEN-NATIVE-001` in the manifest).

## Resolved (July 21, 2026)

Five PrimeNG variables were confirmed **exact aliases** of existing `--ps-*` semantic tokens by
tracing `packages/primeng-preset/src/preset.ts`'s source mapping (each PrimeNG semantic value is
literally assigned from the same `packages/tokens` semantic value that feeds the `--ps-*`
variable) and independently verified via computed-style comparison in both light and dark mode:

| PrimeNG variable | Replaced with | Verified via |
| --- | --- | --- |
| `--p-content-background` | `--ps-surface-card` | `preset.ts`: `content.background = semantic.surfaceCard`; live computed-style match in light (`#ffffff`) and dark (`#0f172a`) |
| `--p-content-border-color` | `--ps-surface-border` | `preset.ts`: `content.borderColor = semantic.surfaceBorder` |
| `--p-text-color` | `--ps-text-primary` | `preset.ts`: `text.color = semantic.textPrimary`; live computed-style match in light (`#0f172a`) and dark (`#f8fafc`) |
| `--p-text-muted-color` | `--ps-text-secondary` | `preset.ts`: `text.mutedColor = semantic.textSecondary` |
| `--p-primary-color` | `--ps-primary-background` | `preset.ts`: `primary.color = semantic.primaryBackground` |
| `--p-border-radius-md` | `--ps-radius-md` | `tokens.css`: `--p-border-radius-md: var(--ps-radius-md)` — a direct, literal alias in the generated output |

These swaps were applied across every native/mixed component that used them
(`ps-card`, `ps-checkbox`, `ps-dialog`, `ps-empty-state`, `ps-form-section`, `ps-input`,
`ps-page-header`, `ps-paginator`, `ps-popover`, `ps-progress`, `ps-select`, `ps-skeleton`,
`ps-status-card`, `ps-table`, `ps-toast`, `ps-tooltip`). `ps-empty-state`, `ps-skeleton`, and
`ps-table` are now fully clean (`tokenBoundary: 'public'`); the manifest's own generation-time
validator (which scans source for `--ps-*`/`--p-*` usage and cross-checks the declared
`tokenBoundary`) confirms every entry's classification now matches reality.

Because each swap is a value-identical alias, this remediation carries no visual-regression risk —
confirmed by direct computed-style comparison in the browser, not just by trusting the source
mapping.

## Still open (no verified safe swap exists)

These remaining `--p-*` usages were investigated and found to have **no confirmed `--ps-*`
alias** — swapping them would require either inventing a new semantic token or accepting an
unverified value change, both bigger decisions than a mechanical remediation pass:

| Component | Usage | Why it's still open |
| --- | --- | --- |
| `ps-card` | `var(--p-card-shadow, none)` | `--p-card-shadow` is actually generated by our own token pipeline (`tokens.css`), not sourced from PrimeNG — despite the `--p-` prefix, it's not a real provider-coupling. Renaming would require adding a new `--ps-card-shadow` token; not done in this pass since it's a naming quirk, not a functional violation. |
| `ps-checkbox`, `ps-input`, `ps-select` | `var(--p-red-600, ...)` for invalid/error text | No confirmed alias to `--ps-danger-color` (`#dc2626`) or `--ps-danger-text` (`#991b1b`) — PrimeNG's `red.600` primitive is not proven to equal either value. |
| `ps-toast` | `var(--p-red-500, #ef4444)`, `var(--p-green-500, #22c55e)`, `var(--p-yellow-500, #f59e0b)` | Same issue — PrimeNG's built-in palette steps for severity colors have no confirmed `--ps-*` equivalent at this exact shade. |
| `ps-toast` | `var(--p-overlay-popover-shadow, ...)` | No `--ps-*` shadow token exists yet for overlay/popover elevation. |
| `ps-paginator` | `var(--p-paginator-nav-button-hover-background)`, `var(--p-paginator-nav-button-hover-color)` | PrimeNG-component-specific token names with no semantic `--ps-*` equivalent; these describe PrimeNG's own paginator sub-parts, not a general surface/text concept. |

## Recommended next step

Adding dedicated `--ps-danger-*` shade tokens (to replace the `--p-red-*`/`--p-green-*`/
`--p-yellow-*` severity-color usages) and a `--ps-overlay-shadow` token would close most of the
remaining list. That is a token-schema addition — a design decision about what those values should
be — not a mechanical alias swap, so it's left as a scoped follow-up rather than done here.
